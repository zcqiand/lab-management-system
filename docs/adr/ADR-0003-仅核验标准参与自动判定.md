# ADR-0003：仅让核验通过的标准要求参与自动判定

- 状态：Proposed
- 日期：2026-07-22
- 决策者：待确认

## 背景

官方附件2只列出检测专项、项目和能力参数，不列单位、限值、判定公式或条款。项目本机 `raw/standards/` 有 61 个唯一标准文件，但资料覆盖不等于已经完成条款提取和人工核对；部分要求是公式、范围、条件表或非数值指标，现有 `comparison + value` 结构不能安全表达。

## 决策

我们决定：

1. 所有标准参数和技术要求带标准版本、源文件哈希、条款/页码和 `verificationStatus`；
2. 从文档提取的候选记录先为 `draft`，人工核对后才可为 `verified`；
3. `evaluateTestResult` 只使用 `verified` 且 `judgmentMode=automatic` 的记录；无匹配、未核验或复杂要求统一返回“需人工判定”；
4. 缺失标准来源保持空值并进入覆盖矩阵，不用经验值、旧项目默认值或附件2文本补齐；
5. 原始标准全文不提交仓库，只提交允许再分发的结构化事实、来源元数据和哈希。

## 备选方案

### 方案 A：导入文档后立即把解析值用于自动判定

被拒绝，因为 PDF/DOCX 表格解析可能错列、标准可能过期，未核对的限值会产生错误合格/不合格结论。

### 方案 B：所有缺失限值统一使用现有种子默认值

被拒绝，因为不同标准、型号、等级和试验条件不能共用默认值，且会掩盖能力目录覆盖缺口。

## 后果

正面：

- 每个自动判定结果都能回溯到标准版本和条款；
- 复杂要求不会被错误地当作简单数值比较；
- 标准资料不完整时系统仍能展示目录，但不会产生假绿判定。

负面（必填）：

- 需要人工逐条核验候选标准记录，标准补齐速度会慢于目录导入速度；
- 很多官方项目在资料补齐前只能保持 disabled/catalogued，不能立即用于正式接样；
- 技术要求模型、自动评定代码和测试数据需要扩展，短期工作量明显增加；
- 仅保存哈希和结构化事实无法替代原文审计，复核人员仍需有权访问受控原始标准文件。
