# ADR-0004：报告名称重做与检测能力收口

- 状态：Accepted
- 日期：2026-07-23
- 决策者：用户

## 背景

检测能力模块（M06）在 ADR-0002 中建立后，主体六个功能已基本成形，但仍有遗留问题：

1. **报告模板是 HTML+`{{标签}}` 的字符串处理**，与 M06 检测能力的实体关联模型格格不入。它面向"按报告类别一份 HTML"展开，无法复用检测能力四类实体关联，也难以追溯报告内容与检测资源的对应关系。
2. **M04 基础数据下报告类别（F01）/报告模板（F02）/标准管理（F03）/检测参数（F04）/合同类别（F10）** 与 M06 检测能力下对应的检测标准（F04）/检测参数（F03）/技术要求（F06）功能重叠或重复，用户希望清理。
3. **型号/规格/等级/牌号** 一直作为 `InspectionTechnicalRequirement` 上的字符串字段（`brand` / `model` / `grade` / `spec`），既无独立实体，也无独立删除保护；同时这些码表的 CRUD 在 M04 下按"报告类别"过滤（报告类别即将废弃），过滤维度失去意义。
4. **报告名称**作为用户概念中"按场景配置检测资源"的实体，尚未存在。它需要挂载四类多对多关联（检测项目 / 检测标准-检测依据 / 检测标准-判定依据 / 检测参数），作为后续报告渲染路径的输入。

## 决策

我们决定：

1. **报告模板 → 报告名称（完全替换）**：
   - 丢弃原 HTML 模板字段（`name` / `content` / `{{标签}}` 渲染逻辑）；
   - 新建 `InspectionReportName` 实体：`id` + `code` + `name` + `sortOrder` + `description`（可选）+ `createdAt` + `updatedAt`；
   - 报告名称隶属 M06 检测能力下，与检测项目、检测标准、检测参数三类实体多对多；
   - 报告 HTML 渲染路径不在本 ADR 范围（后续单独 REQ）；本 ADR 仅确定"按场景挂检测资源"的数据模型。
2. **报告名称 → 检测标准 关联复用 role 枚举**：
   - 新建 `InspectionReportNameStandard`，含 `role: TESTING | JUDGMENT`；
   - 复用现有 `InspectionStandardRole` 枚举，不引入新枚举；
   - 同 (报告名称, 标准) 可存两条不同 role 的记录；
   - UI 上分两个按钮（"关联检测依据"/"关联判定依据"）对应两种 role。
3. **M04 基础数据精简**：F01 报告类别、F02 报告模板、F03 标准管理、F04 检测参数、F10 合同类别整组废弃（状态改为已废弃，行不删号不回收），保留旧 API 读路径供接样单等历史数据查询，写入路径返回 410 Gone。
4. **型号/规格/等级/牌号 提为独立实体**：
   - 新建 `InspectionBrand` / `InspectionModel` / `InspectionGrade` / `InspectionSpec` 四个实体，各占独立 type 文件；
   - M04.F06/F07/F08/F09 CRUD 页保留，列表过滤维度从"按报告类别"改为"按检测专项 (M06.F01)"；
   - 删除保护升级：被 `InspectionTechnicalRequirement` 引用时拒绝删除；
   - `InspectionTechnicalRequirement.brand` / `model` / `grade` / `spec` 字段类型保持 `string`（约定为 code 形态，FK 语义但不引入新 schema 字段）。
5. **类型文件组织**：`src/types/inspection/` 已有"一类一文件"模式，新增 5 个 type 文件：`inspection-report-name.ts` / `inspection-brand.ts` / `inspection-model.ts` / `inspection-grade.ts` / `inspection-spec.ts`；通过 `index.ts` 具名导出。
6. **菜单重挂**：基础数据菜单移除报告类别/报告模板/标准管理/检测参数/合同类别/四码维护（注意：四码维护保留，但菜单项位置可保留）；检测能力菜单新增"报告名称"。

## 备选方案

### 方案 A：保留报告模板作为 UI 配置入口，叠加报告名称

被拒绝，因为报告模板的 HTML 字段无法被检测能力四类实体关联消费，会形成两套真相。

### 方案 B：报告名称关联检测标准用两个独立中间表

被拒绝，因为现有 `InspectionObjectStandard` 已用单表 + role 字段处理同样问题，分两张表会引入新的不一致模式（计算规则/技术要求 也有类似语义）。

### 方案 C：型号/规格/等级/牌号 不提为实体，继续做字符串字段

被拒绝，因为：
- 字符串字段无独立删除保护，技术要求引用一个已下线的型号时无法阻止；
- 缺乏列表过滤维度（M04.F06-F09 当前按已废弃的报告类别过滤）；
- CRUD 页无法独立提供"按检测专项过滤"等业务需求。

### 方案 D：报告名称 隶属 M04 基础数据而非 M06 检测能力

被拒绝，因为报告名称的核心语义是"挂检测能力四类实体"，放在 M04 基础数据会切断与 M06 的关系链，且 M04 正在精简中。

## 后果

正面：

- 检测能力（M06）形成完整闭环：检测专项 → 项目 → 参数 → 标准（含 role 区分）→ 计算规则 → 技术要求 → 报告名称；
- 报告名称作为检测资源的中转实体，向上为后续报告渲染提供配置，向下汇聚四类 M:N 关联；
- M04 基础数据下冗余功能清理，旧 API 读路径保留不影响历史数据；
- 型号/规格/等级/牌号成为独立实体，可独立管理、独立删除保护、独立按检测专项过滤；
- 类型目录结构与 M06 既有模式一致（每类一文件）。

负面（必填）：

- M04.F01-F04/F10 废弃后，旧 REQ（如 REQ-2026-001 / REQ-2026-004）的功能影响表引用这些 ID，语义上"老 REQ 在引用已废弃功能"。需后续 REQ 单独整理（不在本 ADR 范围）；
- `InspectionTechnicalRequirement.brand` / `model` / `grade` / `spec` 字段仍为 `string`，FK 约束只在 MSW 层模拟，db schema 层无外键。后续若上真实数据库需评估迁移成本；
- 报告 HTML 渲染路径尚未实现，报告名称实体暂时无下游消费方；需要后续 REQ 单独处理；
- 5 个废弃功能 + 7 个新增子项 + 5 个 type 文件 + 3 张关联中间表，工作量较大，需分阶段实施；
- M04.F11 计算规则（规划）状态与 M06.F05（规划）平行存在，是否同步迁移未在本 ADR 范围；保留两条路线，避免一次性大改。
