# ADR-0002：建立独立检测能力模块与项目上下文模型

- 状态：Accepted
- 日期：2026-07-22
- 决策者：用户

## 背景

现有 M04 基础数据混合了报告类别、报告模板、标准、参数、技术要求、型号、规格、等级、牌号、合同类别和计算规则。用户要求检测能力形成独立功能模块，并明确其六个组成：检测专项、检测项目、检测参数、检测标准、计算规则、技术要求。

官方附件2存在细骨料、粗骨料、防水卷材等行内分组；用户已明确这些分组直接成为检测项目，代码实体名为 InspectionObject，不存在 InspectionProject/projectId。检测项目与标准、参数均为多对多；标准关联在项目侧还需区分检测依据和判定依据。

## 决策

我们决定：

1. 新增一级模块 `M06 检测能力`，包含 F01 检测专项、F02 检测项目、F03 检测参数、F04 检测标准、F05 计算规则、F06 技术要求；
2. 中文界面使用“检测项目”，TypeScript 使用 `InspectionObject`；
3. 领域层级为 `InspectionSpecialty * ── * InspectionObject`，不存在 InspectionProject、Variant 或 Scheme；多对多通过 `InspectionSpecialtyObject` 中间关联表实现；
4. `InspectionObjectParameter` 表达项目与参数多对多；
5. `InspectionObjectStandard` 表达项目与标准多对多，单条关系角色只能是 TESTING 或 JUDGMENT；
6. `InspectionStandardParameter` 表达标准与参数多对多，并保存条款、方法、单位和试验规则；
7. `InspectionCalculationRule`（原 `CalculationRule`）至少按项目+参数归属，可选限定 TESTING 标准；原文件 `calculation-rule.ts` 改名为 `inspection-calculation-rule.ts`；
8. `InspectionTechnicalRequirement`（原 `TechnicalRequirement`）按项目+参数+JUDGMENT 标准+条件归属；原文件 `technical-requirement.ts` 改名为 `inspection-technical-requirement.ts`；
9. 资质要求级别改名为资质等级 `InspectionQualificationLevel`（QUALIFIED/RESTRICTED），常量数组 `INSPECTION_QUALIFICATION_LEVELS`；
10. 旧 M04 对应功能在 M06 完成迁移后通过后续 tree-change 标记为已废弃，不删除历史 ID；
11. TypeScript 类型按 interface/type 一类一文件组织，通过 index.ts 具名导出。

## 备选方案

### 方案 A：继续重构 M04

被拒绝，因为 M04 还包含报告模板和多个普通码表，检测能力边界无法清晰表达，未来废弃旧模型也困难。

### 方案 B：专项 → 项目实体 → 对象实体 → 参数

被拒绝，因为用户明确不要 InspectionProject/projectId；来源行和运行时项目会重复建模。

### 方案 C：计算规则和技术要求只挂参数

被拒绝，因为同名参数在不同检测项目、标准和条件下可能使用不同算法或限值，容易错误复用。

### 方案 D：不区分检测依据与判定依据

被拒绝，因为试验方法标准和产品判定标准职责不同，接样、报告和单项评定需要分别选择和追溯。

## 后果

正面：

- 检测能力拥有独立功能边界和统一词汇；
- 项目、标准、参数关系可分别维护且支持复用；
- 检测结果计算与单项评定职责分离；
- 后续合同专项和报告项目迁移有明确目标模块。

负面（必填）：

- M06 与旧 M04 在迁移期会短暂重复，必须控制入口和数据源，避免双写；
- 新模块需要 6 个功能、22 个用户操作及多张关系表，改造工作量明显增加；
- CalculationRule 和 TechnicalRequirement 带项目/标准上下文，数据维护复杂度高于参数级规则；
- 旧 M04 只有在全链路迁移完成后才能废弃，不能一次性删除；
- 运行时 InspectionObject 数量可能大于官方 93 个来源行，校验必须同时检查来源行覆盖和对象唯一性。
