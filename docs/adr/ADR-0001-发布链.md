# ADR-0001：采用 tag 触发的 Docker Hub + VPS 发布链

- 状态：已接受
- 日期：2026-07-21
- 范围：lab-management-system 基础设施

## 背景

项目原有 CI 能在分支和 PR 上运行测试、覆盖率、Vite 构建，并验证 Dockerfile 可以构建镜像，但没有把通过验证的版本交付到运行环境。`saas-identity-platform` 已采用版本 tag 触发 Docker Hub 推送和 VPS 重启的发布链，本项目需要复用这个发布模型，同时保留本项目现有的 Docker/nginx 路径。

## 决策

1. 只有推送符合 `v<MAJOR>.<MINOR>-<NNN>` 的版本 tag 才执行发布 job；其它分支和 PR 继续执行测试与本地 Docker 构建验证。
2. 镜像命名为 `${DOCKER_USERNAME}/lab-management-system`，同时推送 `latest` 和版本 tag。
3. VPS 使用 `/home/deploy/lab-management-system/` 和 `lab-management-system` 容器名；容器的 80 端口绑定到 VPS 本机 `127.0.0.1:8062`。
4. 保留根目录 `nginx.conf`、`/usr/share/nginx/html` 和现有 Dockerfile 的多阶段结构，不把参考项目的 `deploy/nginx.conf` 路径直接复制过来；Docker 构建安装依赖时显式使用 npmmirror。
5. Docker Hub 凭据、VPS 地址、用户和 SSH 私钥只通过 GitHub Actions Secrets 提供；CI 通过 SSH action 的环境变量传递 Docker 凭据，部署脚本不从命令行参数读取密码；证书只保留在 VPS。
6. 本次不改变 `package.json` 的 build 脚本、`.harness/stack.json` 的 trace 契约或业务功能树；本次变更是基础设施发布能力，不新增业务 fn-ID。

## 后果

- 发布前必须在 GitHub 配置 `DOCKER_USERNAME`、`DOCKER_PASSWORD`、`VPS_HOST`、`VPS_USER`、`VPS_SSH_KEY`。
- `deploy` 用户加入 Docker 组后具备等同 root 的宿主机控制能力；必须使用专用、受限的 VPS 账号和 SSH key，不能复用个人登录凭据，并应按 VPS 安全策略限制来源和权限。
- 首次部署需要运行 `deploy/setup-vps.sh`、准备证书并安装 VPS 用户的公钥。
- 8062 不能与 VPS 上的其他服务冲突；如果运行环境调整，必须同时修改部署脚本和 nginx vhost 模板。
- 重新推送同一 tag 时，deploy job 通过 GitHub Actions concurrency 不并发执行；回滚可以传入历史版本 tag 重新运行 VPS 部署脚本。
