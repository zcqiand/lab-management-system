## 1. 需求描述

### 用户原话

后续把合同类别改为检测专项 `InspectionSpecialty`，报告类别改为检测项目（代码实体 `InspectionObject`）；检测项目关联检测依据、判定依据和检测参数，计算规则用于计算检测结果，技术要求用于单项评定。

### 理解后的需求描述

新增 M06 后分阶段替换旧 M04 和业务流：

1. `Contract.contractCategory` → `inspectionSpecialtyCode`，数据来源改为 M06.F01；
2. `ReportCategory` → `InspectionObject`，中文 UI 使用“检测项目”；
3. `SampleReceipt.categoryCode` → `inspectionObjectCode`；
4. 接样单的检测依据、判定依据分别来自 InspectionObjectStandard 的 TESTING/JUDGMENT 关系；
5. 检测参数由项目直接关联参数与所选标准关联参数共同约束；
6. 数据录入先使用 InspectionCalculationRule 计算检测结果，再使用 InspectionTechnicalRequirement 做单项评定；
7. M06 迁移完成后，旧 M04 报告类别、标准、参数、技术要求、计算规则和合同类别功能走 tree-change 标记为已废弃，不删除历史 ID；
8. 检测专项与检测项目通过 InspectionSpecialtyObject 多对多关联；项目可归属多个专项，专项包含多个项目。

当前 7 个旧报告类别迁移为 7 个 InspectionObject：水泥、钢筋原材、钢筋机械连接、钢筋焊接、细骨料、粗骨料、混凝土。

## 5. 流程影响

```text
合同选择检测专项
→ 接样选择检测项目
→ 选择检测依据/判定依据
→ 选择检测参数
→ 数据录入
→ 计算规则计算检测结果
→ 技术要求完成单项评定
→ 报告与汇总
```

## 6. 风险与回滚
